## P1-3 SSE 失败检测 heuristic 太脆

**位置**：`app/api/research/multi-agent-stream/route.ts:101-128`

**问题**：用"title 是 Untitled + summary 为空 + innovation 为空 + research_goals 为空"作为"所有 agent 失败"的判断。问题：

- 某些真实边缘 case（论文标题就是 "Untitled"、或无明确 research_goals 的短文）会假阳判为"失败"。
- 某些 agent 失败但 KB 部分成功（KB 内部空字段被兜底填充 "researchkit" tag）的情况会假阴。
- 检测逻辑没考虑"Planner 实际调用了 Reader/Analyzer 但返回内容都是 null"的真实失败模式（只是没解析出有效字段）。

**修复**：用 `result.execution.filter(e => !e.success).length` 作为主要信号，heuristic 只做兜底。

## P1-4 SSE onAgentToken callback 路径有断链

## P1-5 CORS \`\*\` 在 POST/Authorization 端点过宽

**位置**：`app/api/research/fetch-url/route.ts:69-76` + `app/api/research/upload-pdf/route.ts:107-114`

**问题**：两个端点的 OPTIONS 预检都返回 `Access-Control-Allow-Origin: *` + `Allow-Headers: Content-Type, Authorization`。任意域可跨源调用这两个端点，组合 CSRF + 滥用（无认证/无限流）：

- 恶意站点可代用户调 `fetch-url`（SSRF — 但 fetchFromUrl 内部有内网白名单防护）
- 恶意站点可代用户调 `upload-pdf` 让服务端解析恶意 PDF（pdf-parse 2.4.5 已知有历史 CVE）

**修复**：把 CORS Allow-Origin 改成实际部署域名白名单（Vercel 域名 + 自定义域名），或干脆去掉（CORS 头对同源请求无影响，仅当真的需要跨域客户端时才用）。

## P1-6 PDF 上传无 magic-bytes 校验 + pdf-parse 2.4.5 已知风险

**位置**：`app/api/research/upload-pdf/route.ts:46-52` + `package.json:17`

**问题**：仅按扩展名 + MIME 类型校验文件，攻击者可上传 `evil.pdf`（实际是恶意 JS/可执行文件）触发 pdf-parse 解析。pdf-parse 2.4.5 已是相对旧版本（2024 年中），上游维护节奏慢，历史上有多起 PDF 解析漏洞。

**修复**：

1. 在 `parsePdf` 入口加 magic bytes 校验：`if (!buffer.subarray(0, 4).equals(Buffer.from('%PDF'))) throw ...`。
2. 评估迁移到 `pdf-parse-fork` 或 `pdfjs-dist`（更积极维护）。
3. 10MB 限制可保留（合理），但加 `content-length` 在 Next.js edge config 兜底。

## P1-7 SSE Content-Length header 缺失

**位置**：`app/api/research/multi-agent-stream/route.ts:208-214`

**问题**：SSE 响应没显式设 `Transfer-Encoding: chunked`，某些代理（如 Vercel Edge / Cloudflare）会自动加 `Content-Length` 而破坏流式。

**修复**：显式加 `'Transfer-Encoding': 'chunked'` 或确保 Next.js 14 streaming 不被中间件打断。值得本地起一次 next dev，浏览器 devtools 看 Network → response headers。

## P1-8 多维表格/历史 API 无任何鉴权与限流

**位置**：`app/api/history/kc/route.ts` + `app/api/history/cost/route.ts` + `app/api/plugins/install/route.ts`

**问题**：所有 API 路由无任何 auth check / rate limit。任意访客可：

- POST `/api/history/kc` 任意大 payload 填满 `.researchkit-data/kc-history.json`（MAX 10 entries 但 trim 逻辑在 dedupe 之前，恶意不同 payload 可塞满）
- POST `/api/history/cost` 写垃圾
- GET `/api/plugins/marketplace` 拉清单（信息泄露，但当前为静态 mock，影响小）

**修复**：至少加 IP 维度的 rate limit（`@upstash/ratelimit` 或 Vercel 内置），POST 端点加 size limit + JSON schema 校验（zod / valibot）。

## P1-9 planner 失败后 fallbackPlan 跑全量 6 步可能成本爆炸

**位置**：`core/orchestration/planner.ts:75-105`

**问题**：若 LLM 临时 429/timeout，fallbackPlan 跑 Reader+Analyzer+Terminology+KB+Recommendation+Export 6 步全量（每步 1 次 LLM 调用，最少 6 次 + 反思 2 轮 = 10+ 次）。这正好是 LLM 最不稳定的时段，每次都 fallback 等于雪上加霜。

**修复**：加指数退避（1s / 2s / 4s / 8s）后才走 fallback，且 fallback 模式给用户提示"LLM 临时不可用，使用默认 pipeline"。

## P1-10 \`\_\_non_webpack_require\_\_\` 在 production 不可靠

**位置**：`lib/usage-collector.ts:80-93` + `types/global.d.ts:9`

**问题**：D25 用 `__non_webpack_require__` 绕 webpack 静态分析来 require `node:async_hooks`。在 webpack 5 production build 下这个变量会被注入；但若项目迁到 Turbopack（Next.js 15 默认）或 Rspack，这个 trick 会失效，ALS 整个降级为 null，token 归因静默失效。

**修复**：用更稳的 pattern：`import { AsyncLocalStorage } from 'node:async_hooks'` + 在 `next.config.js` 的 `serverExternalPackages` 把 `node:async_hooks` 加入，或在 `experimental.serverComponentsExternalPackages`（v14）配置。把 module-level ALS 实例直接 require 在 server entry，确保 client bundle 永远不进。

# 性能 & 成本建议

## PC1 反思循环 token 成本量化

单次 analysis pipeline 调 LLM 次数：

- Planner：1 次（text mode，流式）
- Reader + Analyzer + Terminology：3 次（并行 JSON mode）
- KnowledgeBuilder：1 次（JSON mode）
- Recommendation：1 次（JSON mode）
- Export：1 次（markdown）
- Reflection：1 次（每轮）
- Replan：1 次（每轮，仅当 reflection 不满意）
- Supplementary steps：1-2 次/轮

**最坏情况**：1 + 3 + 1 + 1 + 1 + 1 + 1 + 2×2 = 13 次 LLM 调用。按 README 平均 13,019 tokens/KC × \$0.0022/KC，对应 deepseek-v4-flash 单价（输入 \$0.14/1M、输出 \$0.28/1M），单次 KC 成本约 \$0.002-0.005。OK，2-5 倍安全余量（README 报价 \$0.0022）。

**但**：若反思循环触发 2 轮（MAX_ITERATIONS=2），最坏路径上 13 次调用 → \~26K tokens / \$0.005。是否值得 1/13 概率触发的 marginal 改善 KB 质量？可考虑把 MAX_ITERATIONS 默认 1，可手动调 2。

## PC2 onchain 真实化无 batching / debounce

当前 onchain-export 每次调都跑：IPFS upload + contract mint + tx send 三步，**全串行**，且无 debounce（连点 5 次"发布"会跑 5 次独立流程）。real 模式下三步加起来可能 5-30s，UX 差且链上手续费翻倍。

**修复**：批量发布接口 + UI loading 状态禁用按钮。

## PC3 prompt caching 未启用

每个 Agent 的 system prompt 是固定的（`prompts/reader.ts` 等），重复调用未利用 OpenAI/DeepSeek 的 prompt caching（OpenAI 自动 cache 命中段 50% 折扣，DeepSeek cache 命中约 0.1 倍价）。当前 `OpenAICompatProvider` 没设 `cache_control` 或类似标记。

**修复**：调研 OpenAI/Anthropic prompt cache API（OpenAI 4o 系列支持 `cached` 自动命中），在 Provider 层加 `cacheControl: { type: 'ephemeral' }` 显式标注（Anthropic）或检查响应 usage 的 `cached_tokens` 字段（OpenAI）。

## PC4 First Load JS 126 kB 自报数据

README 自报 126 kB 实际值需 `next build` 后看 `.next/build-manifest.json`。本次 review 因沙箱限制（1 core / 4GB + 1h 限制）未跑 `npm install`，无法独立验证。建议提交前跑一次：

```bash
npm install --production
npm run build
# 看 .next/build-manifest.json 第一个 page 的 First Load JS
```

若实际超过 150 kB，考虑 `dynamic import` LiveThoughts / PluginPanel / CompareTab（demo 阶段不立即用）。
