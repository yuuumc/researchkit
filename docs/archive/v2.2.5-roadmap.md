# ResearchKit OS v2.2.5 — Quality Release Roadmap

> **类型**：Quality Release — 不加功能，专注质量
> **位置**：v2.2 → v2.3 之间
> **目标**：让 v2.2 在 hackathon 评审现场零翻车
> **制定日期**：2026-07-21

---

## 一、核心目标

| 维度 | v2.2 现状 | v2.2.5 目标 |
|---|---|---|
| 论文质量覆盖 | 仅测试过 Transformer / BERT | 50-100 篇真实论文回归 |
| Prompt 漂移 | 未系统验证 | 输出一致性测试报告 |
| Token 成本 | 平均 $0.0058/篇 | 降低 20%+ |
| 响应速度 | SSE 首字节 ~3s | SSE 首字节 ≤ 2s |
| UI 细节 | 基本可用 | Accordion + hover + 互动动画打磨 |

---

## 二、5 天冲刺计划（D17-D21）

### D17 — 测试套件搭建
**目标**：从 0 到 1 建立可重复运行的回归测试基础设施

**交付物**：
- `fixtures/papers/` — 10 篇代表性论文摘要（5 中 5 英，覆盖 ML/NLP/CV/生物/物理 5 个领域）
- `scripts/regression-test.ts` — 自动化跑 + 报告生成（不引入 jest/vitest，原生 node 脚本）
- `scripts/regression-report-template.md` — 报告模板

**验收标准**：
- `npx tsx scripts/regression-test.ts` 可运行（或 npm run test:regression）
- 每篇论文生成 KC，输出 JSON + Markdown 双格式
- 汇总报告：成功率 / 平均 token / 平均耗时 / 失败 case 详情
- 总耗时 ≤ 5 分钟（10 篇 × 30s/篇）

---

### D18 — 英文论文质量专项优化
**目标**：基于 D17 测试结果，修复英文论文的 prompt 漂移

**交付物**：
- 修复 5 个最严重的英文论文质量问题（基于 D17 报告）
- Prompt 调优 diff（仅修改 prompt 文本，不改架构）
- D17 测试套件重跑验证（修复前后对比）

**验收标准**：
- 英文论文 KC 完整度 ≥ 90%（关键字段：title/authors/field/year/summary/methodology）
- 英文论文无中文残留（outputLocale='en-US' 时）
- 修复前后 token 消耗不增加

---

### D19 — Token 优化 + 响应速度
**目标**：降低 demo 成本，提升评审现场流畅度

**交付物**：
- Token 优化：
  - System prompt 压缩（去掉冗余说明）
  - Few-shot examples 精简（保留结构，删除冗余字段）
  - 上下文裁剪（输入 > 8000 字符时摘要而非截断）
- 响应速度：
  - SSE 首字节提前 flush（Planner 完成后立即推送，不等 Reader）
  - 并行 Agent 改为串行（避免 LLM API 限流）

**验收标准**：
- 平均 token 降低 20%+（基线：26802）
- SSE 首字节 ≤ 2s（基线：3s）
- D17 测试套件重跑验证 KC 质量不退化

---

### D20 — UI 细节打磨
**目标**：第一印象提升

**交付物**（按用户偏好：Accordion + hover + 互动动画）：
- 全局：
  - 卡片 hover lift 动画统一
  - 按钮 click ripple 效果
  - 加载状态骨架屏（替代 spinner）
- 首屏：
  - 输入框 focus 时 glow 效果
  - 模式切换 tab 切换动画
- 结果展示：
  - KC 字段 staggered entry（错落入场）
  - Knowledge Graph 节点 hover 高亮路径
- 移动端：
  - 触摸优化（按钮 ≥ 44px）
  - 折叠卡片默认收起

**验收标准**：
- Lighthouse Performance ≥ 80
- 主流浏览器（Chrome / Edge / Safari）兼容
- D17 测试套件无回归

---

### D21 — v2.2.5 tag + Release
**目标**：发布 v2.2.5 Quality Release

**交付物**：

- v2.2.5 tag + GitHub Release
- CHANGELOG.md 添加 v2.2.5 条目
- package.json version → 2.2.5

**验收标准**：

- 全链路冒烟测试通过
- v2.2.5 tag 创建成功
- GitHub Release v2.2.5 发布成功

## 三、关键约束

### 不做的事
- ❌ 不加新功能（v2.3 才做 Onchain 真实化 / 跨会话记忆）
- ❌ 不改架构（保持 v2.2 的 Plugin System 三层架构）
- ❌ 不引入新依赖（避免 jest/vitest 等测试框架）
- ❌ 不改 API 接口（保持向后兼容）

### 必须做的事
- ✅ 每个 PR 独立合并，不攒多天
- ✅ 每天冒烟测试
- ✅ 修复前后用 D17 测试套件对比
- ✅ v2.2.5-rc 冻结点：D21 上午，下午若发现 bug 则回退到 v2.2

## 五、风险与应急

### 风险 1：LLM API 限流
- **概率**：中（10 篇论文 × 6 agent = 60 次调用）
- **应对**：D17 脚本加入 rate limit（每篇间隔 2s）
- **应急**：切到备用 provider（OpenRouter / Groq）

### 风险 2：测试发现严重 bug
- **概率**：中高（首次系统测试）
- **应对**：D18 专项修复
- **应急**：若 D18 修复不完，D19-D20 顺延，v2.2.5 推迟 1 天

### 风险 3：UI 打磨超时
- **概率**：低
- **应对**：D20 严格按优先级（P0 完成即可发布，P1/P2 可推到 v2.3）
- **应急**：若 D20 超时，直接用 v2.2 的 UI 发布 v2.2.5
