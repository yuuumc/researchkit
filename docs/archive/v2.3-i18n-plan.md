## 一、问题诊断

### 当前 v2.2.6 实际混杂情况

| 层 | 写死位置 | 现状 | 问题 |
|---|---|---|---|
| UI 框架 | `components/settings/SettingsContainer.tsx:14-18` | 全英文写死 | `Provider / Prompt / General / Cost / About` 切语言不变 |
| Tab label | `components/settings/SettingsContainer.tsx:14-18` | 全英文 | 不响应任何语言设置 |
| Output Language 下拉 | `components/settings/tabs/GeneralTab.tsx:28-35` | 中英混杂 | `Auto — 跟随源语言` / `中文 (简体)` / `English (US)` |
| Preset label | `config/presets.ts:33-95` | 英文写死 | `Academic` / `Beginner` / `Developer` |
| Preset description | `config/presets.ts:35-95` | 中文写死 | `学术研究者视角 — 严谨、方法学导向` |
| Help 提示文案 | `components/settings/tabs/GeneralTab.tsx:111,116,134,140` | 中英混杂 | `Prompt 三层架构 + 角色注入` + `Persona` |
| KC 字段标签 | `lib/ui-labels.ts` | 已切换,但绑定错对象 | `getLabels(language, locale)` 用的是 **输入语言**(KC.language),不是 UI 语言 |
| Settings UI 标签 | `components/settings/tabs/ProviderTab.tsx` 等 | 全英文写死 | 不响应任何语言设置 |

### 根本问题

`outputLocale`(原 Output Language)同时承担了 **3 个不同职责**:

1. **UI 显示语言**(应该是独立的 Application Language)
2. **LLM 输出语言**(真正的 Output Language)
3. **Preset 描述语言**(应该跟随 UI 语言)

这是为什么"切到中文,UI 还是英文"的根本原因。

## 二、目标架构(4 层语言分离)

按 [语言区分.txt](file:///C:/Users/Administrator/Desktop/hack/语言区分.txt) 建议:

```
┌─────────────────────────────────────────┐
│  Application Language  (UI / Help / Tip) │ ← 新增
├─────────────────────────────────────────┤
│  Output Language       (KC 最终输出)     │ ← 保留并解耦
├─────────────────────────────────────────┤
│  Prompt Language       (锁死 English)   │ ← 新增(只读)
├─────────────────────────────────────────┤
│  Auto Translate        (解释/推荐翻译)  │ ← 新增
└─────────────────────────────────────────┘
```

### 4 层职责定义

| 层 | 控制范围 | 选项 | 默认值 | 存储 |
|---|---|---|---|---|
| Application Language | UI / Help / Tooltip / Tab label / Preset 描述 | Auto / 中文 / English / 日本語 | Auto(跟随浏览器) | localStorage `researchkit:app-locale` |
| Output Language | KC 最终输出语言(影响 LLM prompt 的 locale directive) | Auto / 中文 / English / 日本語 / ... | Auto(跟随输入) | localStorage `researchkit:output-locale`(原 outputLocale) |
| Prompt Language | 发送给 LLM 的 prompt 本身 | English(固定) | English | 只读,不允许修改 |
| Auto Translate | Explain / Recommend / Chat 模块的回复是否翻译为 Application Language | On / Off | On | localStorage `researchkit:auto-translate` |

### 解耦后的关系

- **Application = 中文 + Output = English** → UI 全中文,KC 输出英文(允许,典型"看中文 UI 写英文论文"场景)
- **Application = English + Output = 中文** → UI 全英文,KC 输出中文(允许,典型"英文 UI 看中文论文"场景)
- **Application = Auto** → 跟随浏览器 `navigator.language`,默认 fallback 中文

### D36 — i18n 基础设施搭建

**目标**:建立 `src/locales/` 目录 + 翻译文件 + i18n context

**交付物**:

1. `src/locales/` 目录结构:
   ```
   src/locales/
   ├── zh-CN/
   │   ├── common.json      (按钮/菜单/通用词)
   │   ├── settings.json    (Settings 5 个 tab 的所有文案)
   │   ├── preset.json      (5 个 Preset 的 label + description)
   │   ├── home.json       (首页所有文案)
   │   ├── agent.json      (AgentTimeline / SmartSuggestion / Chat / Explain)
   │   └── export.json     (Markdown / Obsidian / Compare tab)
   ├── en-US/
   │   └── (同结构)
   └── ja-JP/
       └── (v2.4 再补,v2.3 先留 stub)
   ```

2. `lib/i18n.ts` — i18n 核心:
   ```typescript
   export type AppLocale = 'zh-CN' | 'en-US' | 'ja-JP' | 'auto'
   export function t(key: string, params?: Record<string, string>): string
   export function useAppLocale(): AppLocale  // React hook
   export function detectBrowserLocale(): AppLocale
   ```

3. `components/I18nProvider.tsx` — React Context:
   - 包裹整个 app
   - 监听 localStorage `researchkit:app-locale`
   - 跨标签页同步(languagechange 事件)

4. `lib/user-preferences.ts` schema 扩展:
   ```typescript
   interface UserPreferences {
     preset: PresetId           // 已有
     outputLocale: Locale | 'auto' // 已有,职责收窄为只控制 KC 输出
     appLocale: AppLocale        // 新增:Application Language
     autoTranslate: boolean      // 新增:是否自动翻译 Explain/Recommend 输出
     promptLanguage: 'en-US'    // 新增:只读,永久 'en-US'
   }
   ```

**验收**:
- `npm run dev` 启动无报错
- 现有功能全部正常(因 fallback 到 en-US)
- `t('settings.tabs.provider')` 返回 `'Provider'`(en-US)或 `'模型'`(zh-CN)

## 五、风险与应急

### 风险 1:KC 字段标签行为变化影响老用户
- **概率**:中(原"输入中文 → 标签中文"行为会变)
- **应对**:D38 在 release notes 中明确说明行为变化
- **应急**:若用户反馈强烈,加一个兼容开关 `kcLabelFollowsInput`(默认 false)

### 风险 2:翻译遗漏导致部分文案仍英文
- **概率**:高(文案分散在 30+ 文件)
- **应对**:D40 用脚本扫描所有硬编码英文字符串(`grep -rn "[A-Z][a-z]\+:" components/`),确保全部走 i18n
- **应急**:遗漏的文案 v2.3.1 hotfix 补齐

### 风险 3:Auto Translate 影响 LLM 输出质量
- **概率**:低(只在 prompt 末尾追加一句)
- **应对**:D39 测试时对比 Auto Translate On / Off 的 KC 质量
- **应急**:若质量下降,Auto Translate 默认改为 Off,用户手动开启

### 风险 4:5 天工期不够
- **概率**:中(30+ 文件改造)
- **应对**:D36-D38 是核心,D39-D40 可压缩
- **应急**:若 D38 卡住,D39 的 Auto Translate 推到 v2.4,D40 只做基础测试发布

## 七、技术决策记录

### 为什么不引入 next-intl / react-i18next?

1. **零重型依赖原则** — 与 v2.2.5 regression test 不引入 jest/vitest 一致
2. **Bundle 体积** — next-intl 增加 ~20KB,react-i18next 增加 ~15KB,自实现 < 1KB
3. **学习成本** — 自实现的 `t(key)` API 与主流库几乎一致,迁移成本低
4. **可控性** — Auto Translate 等自定义逻辑更容易集成

### 为什么 KC 字段标签改为跟随 Application Language?

原行为:输入中文论文 → KC 字段标签中文;输入英文论文 → KC 字段标签英文

问题:用户看不懂英文却被迫看英文标签(因为输入是英文论文)

新行为:Application = 中文 → KC 字段标签永远中文,无论输入什么语言

理由:**UI 是给人看的,应该用用户能懂的语言**,KC 内容(论文标题/作者/摘要)跟随 Output Language 是另一回事

### 为什么 Prompt Language 锁死英文?

1. **LLM 训练数据英文占比最高**,英文 prompt 在所有模型上效果最稳定
2. **避免 prompt 翻译引入歧义**(尤其术语 / few-shot 示例)
3. **硬约束已规定"prompt 文本完全照搬"**,锁死英文与硬约束一致
