## Phase 1 — P0 安全 + 文档真实性

`commit 317f6d1` · `lib/onchain-modes.ts` + `core/plugins/onchain-export.ts` + docs

- **onchain mode 安全 fallback**：`resolveOnchainMode()` 在用户选 `real` mode 时强制 fallback 到 mock + console.warn，避免未实现的 6 个接口抛 NotImplementedError 导致崩溃
- **文档真实性**：onchain-export 插件 description 明确标注 "v2.2 demo 不会真实广播"，去除 "真实发布" 等误导文案
- **凭证泄露防护**：API 错误响应不再回显完整 stack trace（避免泄露内部路径/密钥片段）

## Phase 2 — Vercel 部署修复

`commit 5c83a71` + `242d0e2` + `bcd743c` · `lib/tools/{memory,filesystem}.ts` + `lib/persistence/{kc,cost}-history-server.ts` + `app/api/research/*/route.ts`

### Vercel 只读文件系统
- **问题**：Vercel serverless 运行在 `/var/task/` 只读 fs 上，写入 `.researchkit-memory.json` 报 `EROFS: read-only file system`
- **修复**：4 个持久化模块检测 `VERCEL` 环境变量时改用 `/tmp/` 作为存储路径
- **影响文件**：`lib/tools/memory.ts` / `lib/tools/filesystem.ts` / `lib/persistence/kc-history-server.ts` / `lib/persistence/cost-history-server.ts`

### JSON Parse 诊断增强
- **问题**：`/api/research/multi-agent-stream` + `/api/research/multi-agent` 报 `SyntaxError: Expected property name or '}' in JSON at position 1`，错误信息不可定位
- **修复**：`await request.json()` 改为 `await request.text()` + 手动 `JSON.parse`，失败时返回 HTTP 400 + 完整请求体片段 + 字符位置

## Phase 3 — P1 安全 + UX 基线

`commit 9bdabf4` + `24d672d` · 新增 `lib/cors.ts` + `lib/rate-limit.ts` + `config/orchestration.ts` + 8 个 API 路由

### P1-5 CORS 白名单
- 新增 `lib/cors.ts` — `resolveAllowedOrigin()` 三层策略：
  - same-origin（默认不发 ACAO header）
  - 开发环境允许 localhost
  - Vercel preview 允许 `*.vercel.app`
  - 生产环境通过 `CORS_ALLOW_ORIGINS` 环境变量配置
- 10 个 API 端点统一接入 `corsHeaders()` + `handleOptions()`

### P1-6 PDF Magic Bytes 校验
- `/api/research/upload-pdf` 检查文件头前 5 字节是否为 `%PDF-`（0x25 0x50 0x44 0x46 0x2d）
- 防止 `.exe` / `.html` 改名上传

### P1-8 Rate Limit
- 新增 `lib/rate-limit.ts` — 内存 Map 计数（每 serverless 实例独立，弱一致但防刷够用）
- 5 个 LLM 端点接入：KC 10/min、PDF 5/10min、Batch 3/10min
- `getClientIp()` 从 `x-forwarded-for` / `x-real-ip` 提取 IP
- 定期清理过期 bucket 避免内存泄漏

### P1-2 onchain 导出二次确认
- `PluginPanel.handleExport()` 在 `plugin.meta.id === 'onchain-export'` 时弹 `window.confirm`
- 防止误触链上 broadcast（即便当前是 mock，未来切 real 时已是必须层）

### P1 Planner 指数退避
- `core/orchestration/planner.ts` — LLM 调用失败时 1s → 2s 重试，最多 3 次
- 全部失败再走 `fallbackPlan`，避免单次网络抖动直接降级

## Phase 4 — P2/P3 工程化清理

`commit 49827be` + `2696bf8` · `config/orchestration.ts` + 6 个文件

### P2-1 Magic Numbers 集中
- 新增 `config/orchestration.ts`，所有阈值统一管理：
  - `MAX_ITERATIONS = 2` / `MAX_SUPPLEMENTARY_STEPS = 2`
  - `TOKEN_FLUSH_INTERVAL_MS = 30` / `MIN_CONTENT_LENGTH = 50` / `MAX_CONTENT_LENGTH = 50000`
  - `MAX_PDF_SIZE_BYTES = 10 * 1024 * 1024`
  - `RATE_LIMIT_KC` / `RATE_LIMIT_PDF` / `RATE_LIMIT_BATCH`
  - `PLANNER_MAX_ATTEMPTS = 3` / `PLANNER_BASE_DELAY_MS = 1000`
- 每个常量注释说明 trade-off

### P2-6 URL 修复
- `onchain-export.ts` + `plugin-marketplace-server.ts` 的 `externalApis` 从假 URL 改为真实 host
  - `xlayer.okx.com` → `xlayerrpc.okx.com`
  - `api.ipfs.com` → `api.pinata.cloud` + `api.web3.storage`

### P2-7 Dead Code 删除
- `core/plugins/registry.ts` 删除 `initBuiltinPlugins()`（实际注册走 PluginPanel.tsx 客户端）

### P2-8 TODO 标记
- `core/orchestration/executor.ts` 的 `payloadToContext` 函数前加 TODO 注释

### P3-2 getLabels 注释
- `lib/parser.ts` 的 `getLabels` 函数前加注释说明与 `ui-labels.ts` 的 `getKcFieldLabels` 语义不同

### P3-6 LiveThoughts rAF 节流
- `components/LiveThoughts.tsx` autoscroll 改用 `requestAnimationFrame`
- 原因：长流（Planner 1k+ token）时每个 token 都触发 `scrollTop=scrollHeight` 会 reflow
- 策略：rAF 合并多个 token 更新为一帧一次滚动（比 `useLayoutEffect` 安全，不阻塞 paint）
