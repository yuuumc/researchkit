### D19 — Token 优化 + SSE 首字节优化(从 90% → 100%)

[PR #25](https://github.com/yuuumc/researchkit/pull/25)

**Token 分布分析** — `scripts/analyze-tokens.ts`:
跑 1 篇 fixture dump `per_agent_usage`,揭示 token 热点:
```
Total tokens: 11779
Terminology       5988 tokens  3 calls  $0.0011  (51%)
Planner           3207 tokens  1 calls  $0.0005  (27%)
Reflection        1301 tokens  1 calls  $0.0002  (11%)
Recommendation    1283 tokens  1 calls  $0.0002  (11%)
```

**Token 优化** — `core/orchestration/workflow.ts`:
```typescript
// D19 Token 优化:截断 supplementary_steps 到最多 MAX_SUPPLEMENTARY_STEPS 个
const MAX_SUPPLEMENTARY_STEPS = 2
const truncatedSteps = replanResult.supplementary_steps.slice(0, MAX_SUPPLEMENTARY_STEPS)
```
每个 supplementary agent re-run 成本 ~2000 tokens,capping at 2 bounds worst-case Reflection loop cost。**保留 Replan prompt verbatim**(硬约束),仅后处理 LLM 输出。

**SSE 首字节优化** — `app/api/research/multi-agent-stream/route.ts`:
```typescript
// D19 SSE 首字节优化:立即发送 ping 事件强制 flush 连接
send('ping', { ts: Date.now() })
await new Promise(resolve => setTimeout(resolve, 0))
// 然后才调 coordinate()
```
原因:Next.js / Node.js HTTP 可能缓冲响应直到第一个 `await` 才 flush,导致客户端等待 Planner 完成(~3s)才收到第一个字节。修复后在 `coordinate()` 之前发送 `ping`,建立 SSE 连接并 flush 头部。

**结果**:10/10 (100%) 成功率!BERT en-002 之前失败现在通过(Reflection loop 触发了 supplementary Analyzer,补足了 methodology 字段)。

## 五、Known Limitations(v2.3 改进)

按 v2.2.5 roadmap 风险与应急策略,**P0 完成即可发布,P1/P2 推到 v2.3**:

### P1 推迟
- **Lighthouse Performance ≥ 80** 未达成 — dev 模式含 source map + HMR client 失真,推迟到 ChainHack v2.3 production build 后严格测
- **Token 降幅未达 20% 目标** — supplementary_steps cap 只在 Reflection loop 触发时生效(大多数 fixture 一次 satisfied);Prompt 文本受硬约束不能改,更深的 token 优化(schema-aware Terminology batching)推迟到 v2.3

### P2 推迟
- **Knowledge Graph 节点 hover 路径高亮** — 当前 `kg-branch-hover` 是单 branch,完整 DAG 路径高亮推到 v2.3
- **Onchain Export 仍是 Demo Mode**(v2.2 已知,v2.3 改)
