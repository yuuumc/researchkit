## 核心问题

「载入示例」是 hackathon 评委最常点击的按钮，触发完整 7-stage 多 agent 流水线。实测 stage 3（Concepts Extracted）占 60-80% 总耗时，因为 `executePlan` 在 group 1 并行跑 Reader(30K) + Analyzer(20K) + Terminology(20K) 三次 LLM 调用、Recommendation 还要再串行 2 次 LLM + 最多 12 次外部 HTTP 搜索（arXiv + SemanticScholar，均无超时），全栈总耗时通常 **30-90 秒**。

Vercel Hobby plan serverless 60s 硬超时下，cold start 时极易触发 58s `Promise.race` 超时保护，返回 "生成超时：pipeline 在 58 秒内未完成"，对 demo 是致命的。

---

## 优化方案（方案 C：示例缓存 + 演示性回放引擎）

### 思路
示例内容固定（`EXAMPLE_FIXTURE`），输出对同一 model/locale/preset 也是确定的 → 预计算一次录制，部署时携带，命中即回放，**直接消灭 stage 3 等待**。

### 三层缓存
1. **进程内 Map**（热路径，零 IO；同实例重复点击零延迟）
2. **仓库 `fixtures/example-cache/<hash>.json`**（部署即带，Vercel 也能命中）
3. **运行时 `.researchkit-cache/<hash>.json`**（dev / 自托管；Vercel 只读 fs 静默失败）

### Cache key 严格门控
`sha256(normalize(content)) + providerType + model + outputLocale + preset` — 任一不一致 → miss（避免错语言/错模型的输出被误用）。仅当 content 命中 `EXAMPLE_FIXTURE` 时才查缓存（其他输入永不命中，永不录制）。

## Hotfix — 回放耗时异常

### 问题
线上冒烟发现回放耗时 **40-47 秒**（预期 5-7s），cacheHit=true 但 wall time 远超预期。

### 诊断
- 用 tsx 内联脚本直接调 `replayExample()` 测得 wall=47306ms, totalSleep=38858ms
- token gap 同步计算：678 token × minGap=5 计算总 gap 应为 8145ms（预期约 8s）
- **678 × 50ms = 33900ms + stage dweels ≈ 38858ms（与实测精确吻合）**

### 根因
`lib/example-cache.ts` 中 `DEFAULT_REPLAY_OPTIONS` 硬编码 `minEventGapMs: 50`，而非引用 config 常量 `EXAMPLE_REPLAY_MIN_EVENT_GAP_MS`（值为 5）。config hotfix（commit `316c7d7` 50→5）对回放引擎无效。

### 修复
`commit 21b950b` — `DEFAULT_REPLAY_OPTIONS` 改为引用 `config/orchestration.ts` 的 5 个常量（`EXAMPLE_REPLAY_TIME_SCALE` / `_MIN_EVENT_GAP_MS` / `_MAX_EVENT_GAP_MS` / `_STAGE_MIN_DWELL_MS` / `_TAIL_MS`），单一事实源。

### 验证
| 指标 | 修复前 | 修复后 | 目标 |
|---|---|---|---|
| 本地 tsx wall | 47306ms | 15675ms | <58s ✓ |
| 生产冒烟 #1 | 40965ms | **9778ms** | <58s ✓ |
| 生产冒烟 #2 | — | **14448ms** | <58s ✓ |

**4-5x 提速**，远低于 Vercel 58s 硬约束，demo 节奏仍保留 token 流渐进效果。
